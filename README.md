# Adicot HVAC Loads PDF Pipeline — Flask App

A local web UI for converting Design Master HVAC HTML exports into the three signed PDF deliverables.

## Running locally (laptop / office)

```bash
pip install -r requirements.txt
python app.py
```

Open <http://localhost:5000>. The app binds to `0.0.0.0:5000`, so other devices on the same LAN can also reach it via `http://<your-laptop-ip>:5000`.

For local dev, no password is needed by default. To enable basic auth even locally:

```bash
APP_PASSWORD=mySecretPassword python app.py
```

## Deploying to Render (publicly accessible URL)

This puts the app at something like `https://adicot-hvac.onrender.com`, which you can link to from the Adicot Wix site.

### One-time setup

1. **Push the code to a GitHub repo.** Private repo is fine. Render reads from GitHub on every deploy.

2. **Create a Render account** at <https://dashboard.render.com>. Sign in with GitHub.

3. **Click "New +" → "Blueprint"** and select your repo. Render reads `render.yaml` and proposes the service. Click "Apply".

4. **Set the password.** In the new service's dashboard, go to **Environment** and set:
   - `APP_PASSWORD` — pick a shared password engineers will type (username is always `adicot`)

   `SECRET_KEY` and `JOBS_DIR` are auto-configured by the blueprint.

5. **Wait ~3 minutes** for the first build. Subsequent deploys are faster.

### What you get

- Public URL like `https://adicot-hvac.onrender.com`
- 1 GB persistent disk at `/var/data/jobs` so the past-jobs list survives restarts
- Free tier: spins down after 15 minutes of inactivity (first request after that takes ~30 seconds to wake up). Upgrade to the $7/mo Starter plan for always-on.

### Updating after code changes

```bash
git commit -am "your change"
git push
```

Render auto-deploys on push to the main branch.

### Linking from Wix

In the Wix editor:
1. Add a button (or text link) anywhere on the site
2. Set its action to "Link to web address"
3. Use the Render URL: `https://adicot-hvac.onrender.com`
4. Check "Open in new tab"

Engineers click the link, get the HTTP basic auth prompt (username: `adicot`, password: whatever you set), and use the tool.

## How it works

1. **New Job** page: upload the Design Master HTML, enter the project address, optionally override zone display names / tonnage / supply CFM / merge zones.
2. **Results** page shows downloads for the three PDFs (Ventilation Schedule, Air Balance, Load Summary) plus a console-style preview of all three tables for sanity check.
3. **Past Jobs** page lists every job by project name and lets you re-open or delete.

## Files

- `app.py` — Flask entry point (routes, request handling, auth)
- `hvac_pipeline.py` — Pipeline logic (parsing, calc engine, PDF rendering, console preview)
- `templates/` — Jinja2 HTML templates
- `static/style.css` — UI styling
- `requirements.txt` — Python dependencies
- `Procfile` — Tells Render/Heroku to use gunicorn
- `runtime.txt` — Pins Python version
- `render.yaml` — Render blueprint (one-click deploy config)

## Environment variables (production)

| Variable | Required | Purpose |
|---|---|---|
| `APP_PASSWORD` | yes (prod) | Shared password for HTTP basic auth. If unset, auth is disabled. |
| `SECRET_KEY` | recommended | Flask session key. Auto-generated by render.yaml. |
| `JOBS_DIR` | optional | Workspace path. Defaults to `./jobs`. On Render, set to `/var/data/jobs`. |
| `PORT` | auto | Render sets this automatically. |

## Adjusting the pipeline

To tweak parsing, calc logic, or PDF formatting, edit `hvac_pipeline.py`. After edits, push to GitHub and Render auto-deploys (or restart `python app.py` for local dev).

## What this doesn't do

- **Real user accounts.** It's one shared password for everyone. If you need per-engineer logins, switch from basic auth to Flask-Login.
- **Concurrent users at scale.** Gunicorn with 2 workers handles 2 simultaneous PDF generations. Plenty for a small firm, not enough for hundreds of users.
- **Backups.** The Render persistent disk is reliable but isn't backed up by Render automatically. If you care about job history, add a periodic `tar` to S3 or similar.
